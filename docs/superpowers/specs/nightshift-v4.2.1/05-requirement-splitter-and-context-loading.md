# 05. Requirement Splitter And Context Loading

## Purpose

V4.2.1 inherits the V4.2 splitter and context-loading model and adds the missing normalization pipeline from splitter proposal to immutable `IssueContract`.

Base source:

- [../nightshift-v4.2/05-requirement-splitter-and-context-loading.md](../nightshift-v4.2/05-requirement-splitter-and-context-loading.md)

Where this document adds rules, it supersedes the V4.2 text.

## Proposal vs Contract Rule

The splitter does not emit a final immutable contract directly from its first model output.

It emits a proposal that must pass through normalization.

V4.2.1 distinguishes:

- `SplitterProposal`
  A suggested issue candidate shown to a human
- `IssueContract`
  The normalized immutable issue contract stored after approval

## Proposal Output

A `SplitterProposal` should contain at least:

- title
- description
- suggested kind
- suggested allowed paths
- suggested validation strategy
- suggested acceptance
- notes on risk or dependency

This is intentionally not yet a complete `IssueContract`.

## Normalization Pipeline

After human review, NightShift must run a normalization step before writing a final `IssueContract`.

The normalization step is responsible for resolving all required contract fields.

Suggested pipeline:

1. generate `SplitterProposal`
2. present proposal to the human
3. human confirms, edits, deletes, or adds proposals
4. normalize approved proposal into `IssueContract`
5. reject normalization if required fields remain unresolved
6. persist the final immutable contract

## Required Field Resolution

The normalization step must resolve:

- `issue_id`
- `title`
- `kind`
- `priority`
- `goal`
- `allowed_paths`
- `forbidden_paths`
- `verification`
- `test_edit_policy`
- `attempt_limits`
- `timeouts`

Suggested resolution sources:

- `issue_id`
  generated by NightShift at contract creation time
- `goal`
  explicit human-approved statement of the issue objective, derived from the proposal and refined during review
- `priority`
  explicit human choice, backlog metadata, or `nightshift.yaml.issue_defaults.default_priority`
- `allowed_paths`
  proposal candidate paths confirmed or narrowed during human review
- `forbidden_paths`
  explicit human choice or `nightshift.yaml.issue_defaults.default_forbidden_paths`
- `verification`
  explicit executable commands and pass conditions, never a vague note
- `test_edit_policy`
  `nightshift.yaml.issue_defaults.default_test_edit_policy` with optional human override
- `attempt_limits`
  `nightshift.yaml.issue_defaults.default_attempt_limits` with optional human override
- `timeouts`
  `nightshift.yaml.issue_defaults.default_timeouts` with optional human override

## Default Policy Source Rule

If normalization relies on project defaults, the canonical source is:

- `nightshift.yaml.issue_defaults`

Defaults used for normalization must be explicit and repository-local.

They must not live in hidden code constants if they affect the emitted immutable `IssueContract`.

## No Silent Guessing Rule

If normalization cannot resolve a required field concretely, the system must not silently invent one for an execution issue.

Instead it should:

- keep the item as `planning`
- keep the item as `repro`
- keep the item as `investigation`
- or mark the proposal for human refinement

## Execution Issue Admission Consequence

An item may be normalized into `kind=execution` only when:

- executable validation is concrete
- path constraints are plausible
- policy defaults have been resolved into explicit contract fields
- a human has approved the final execution-ready shape

This is how the splitter output stays compatible with the V4.2.1 immutable contract model.
